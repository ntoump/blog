After spending a lot of time experimenting with different hyperparameter options, I came to the realization that I should perhaps leave aside the Random Forest Regressor for the time being, as I did not want to dedicate anymore time studying related literature with seemingly marginal improvements.

Instead, I decided to try out a new type of algorithm; one that is not widely used for Tabular data, and that has quite a few complications both in its training and its tuning in order to achieve a decent model.

That is, of course, Neural Networks.

Neural Networks

I will again be using the fastai library, which adds functionality on top of PyTorch.

[I have been using fastai in every project displayed thus far in this blog. That is mainly because it provides a reliable and resilient API that handles low-level details, thereby allowing the engineer to focus on high-level abstraction, which can result in less implementing and more experimenting - and in case of learning and researching, the more experimenting the merrier.]

Data

I generally followed the same data preprocessing routine (which you can find here) in this implementation as well, with a few largely unimportant exceptions (such as removing redundant columns and swapping the classification of one or two (depending on the model attempted) column names from Categorical to Continuous).

splits = RandomSplitter(valid_pct=0.2)(range_of(df))
to = TabularPandas(df, 
                   procs, 
                   cat, 
                   cont, 
                   dep_var, 
                   splits=splits, y_block = CategoryBlock)

NNs being NNs require special treatment regarding the format in which the data is inputted in the model. That's why it is necessary to create DataLoaders batches, before constructing the fastai-powered Learner.

dls = to.dataloaders(bs=64)

Classification rather than Regression

From now on I will be treating the competition as a classification problem, I thereby formatted the data's labels as categories _(yblock=CategoryBlock). This is the main difference between the data used thus far and the new TabularPandas instance above.

[Later on prepare to have that approach challenged, as I wanted to play around with distinct but not necessarily contrasting paths.]

Learner & Training

Wherever it is not specified otherwise, consider that the only parameters changed are the ones explicitly mentioned; in other words, all the default values were left as is unless otherwise indicated.

First model 

learn = tabular_learner(dls, metrics=accuracy)
Training

A few initial notes:

  • The model is training unusually fast. But then again, it is training on Tabular data, and, even more so, only a handful of them.
  • Its accuracy is already decent.
  • One would expect that the model is going to start overfitting fairly soon, as I have not adjusted the training learning rate (nor will I do so in this routine).

As expected, the model started overfitting early on. It hit, however, an accuracy of ~84.2% right before that. Perhaps with a few tweaks the next one can become even better.

Review

Inspecting a few random results, we see that the values of the columns Survived and _Survivedpred are almost identical. The labels and the predictions are, therefore, in this small sample fairly well matched, even after training for way too long for the model not to overfit.

Targeted Predictions

I should also mention that it is possible to get the model's predictions on a specified row (or series of rows) of a DataFrame object. An example case is presented below.

For clarity's sake:

  • The variable row contains all the initial row info that the model inputs.
  • clas contains the model's prediction regarding the classification of the entry (in this case, it correctly predicts that the person corresponding to this entry will not survive (0 instead of 1)).
  • probs has the probabilities of each category (in this case we only have two possible prediction outcomes; see here an example of Multi Label Classification using one hot encoding).

(Unsuccessful) Next models

I experimented for quite a while with different NN training routines that I had found useful in the past. Slowly but surely, I started tweaking the hyperparameters as well, always maintaing the same NN architecture.

The results were either approximately the same or worse than the ones presented above. Until I encountered the Breakthrough.

Breakthrough

What I have taken the liberty of naming as such almost exclusively consists of two revelations I had;

  • increasing the batch size (bs) is not always a great idea; decreasing it could, in smaller datasets (ds), prove itself extremely beneficial,
  • the default fastai architecture for Tabular data Learners, while not bad and pretty-much universally applicable, needs to be adapted depending on the peculiarities of each task.

Conclusion

In this short Part, I introduced Neural Networks as a possibly viable (and perhaps an even better) alternative to Random Forests. In continuation, I will elaborate on the Breakthrough, try out different versions, and, probably, proceed to making the second submission on Kaggle.
